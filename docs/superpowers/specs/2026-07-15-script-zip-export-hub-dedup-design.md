# 脚本 ZIP 导出与 Hub 去重安装 — 设计规格

**日期：** 2026-07-15  
**状态：** 已确认设计  
**范围：** Autoforge 桌面端脚本包导出、Hub 下载包身份持久化、重复安装交互与更新

## 1. 背景

Autoforge 已支持从 Hub 下载 ZIP、解压并调用脚本注册表导入。当前每次导入都会生成新的本地 UUID，Hub 请求中的 `hubScriptId` 仅用于日志，因此重复安装会创建新副本。应用也没有面向 Hub 上传场景的脚本 ZIP 导出能力。

本次实现两个能力：

1. 将本地脚本导出为可上传 Hub 的 ZIP，只包含运行所需的项目代码和明确声明的资源，禁止打入已下载依赖和业务数据。
2. 使用 Hub 分配的稳定 `hubScriptId` 识别重复安装；重复时由用户选择更新现有脚本或取消。

## 2. 目标与非目标

### 2.1 目标

- 在脚本卡片操作区提供“导出 ZIP”。
- 自动从入口文件发现静态项目依赖，并生成可审查的导出清单。
- 支持通过 manifest 显式补充无法静态发现的运行资源。
- 强制排除依赖目录、缓存、运行产物、附件、数据库和常见业务数据。
- 将可信 Hub 请求中的 `hubScriptId` 持久化，并保证非空值唯一。
- 重复安装时弹窗选择“更新”或“取消”。
- 更新保留本地用户状态与配置，失败时保留原脚本可用。

### 2.2 非目标

- 不上传 ZIP 到 Hub；本次只生成本地 ZIP 文件。
- 不自动发布、登录 Hub 或管理 Hub 脚本版本。
- 不对动态执行的任意路径做完整程序分析。
- 不把普通本地导入按内容哈希去重。
- 不自动识别旧版本中已经重复导入、但没有 `hubScriptId` 的脚本。

## 3. 核心决策

### 3.1 导出采用依赖图白名单

导出器从 manifest 的入口开始递归解析静态本地依赖，仅将以下内容加入候选清单：

- `autoforge.json`；
- manifest 指定的入口；
- 入口递归引用的工作区内源码或静态资源；
- 包根目录中的 `README.md` 或 `README`；
- manifest 中 `export.include` 显式声明且通过安全校验的文件。

第三方包导入只用于确认依赖边界，不从 `node_modules`、Python 虚拟环境或全局依赖目录复制文件。

### 3.2 Hub ID 只信任安装请求

`hubScriptId` 由 Hub 调用本机桥时提供。桌面端对其做非空字符串规范化后写入本地数据库。ZIP 内即使包含同名字段也不作为身份来源，避免任意包冒充已安装的 Hub 脚本。

导出本地脚本时不要求拥有 `hubScriptId`。Hub 在上传成功后分配 ID，并在后续下载请求中传回该 ID。

### 3.3 重复安装更新现有记录

若 `hubScriptId` 已存在，主进程显示原生确认对话框：

- **更新：** 保留现有本地 `scriptId` 和用户偏好，以新包替换工作区代码并刷新 manifest 派生元数据；
- **取消：** 不修改文件或数据库，并向 Hub 桥返回明确的取消结果。

## 4. 数据模型

`ScriptMeta` 新增可选字段：

```ts
hubScriptId?: string
```

SQLite `scripts` 表新增可空列 `hub_script_id`，并为非空值建立唯一索引。数据库迁移必须兼容现有数据：旧记录的该列均为 `NULL`，多个 `NULL` 不冲突。

仓储层增加按 Hub ID 查询方法。所有写入点统一 trim；空字符串按 `undefined`/`NULL` 处理。唯一索引作为并发安装时的最终一致性保护。

## 5. Manifest 导出配置

脚本 manifest 可选增加：

```json
{
  "export": {
    "include": ["templates/**/*.html", "assets/logo.png"]
  }
}
```

`export.include` 仅用于补充运行必需、但无法由静态依赖分析发现的资源。规则相对于脚本包根目录解析，支持受限 glob；绝对路径、`..`、符号链接目标越界和命中强制排除项的规则均报错或被拒绝，不能覆盖安全策略。

## 6. 导出文件发现

### 6.1 JavaScript / TypeScript

分析以下静态形式：

- ESM `import ... from './x'`；
- ESM `export ... from './x'`；
- 字面量动态导入 `import('./x')`；
- CommonJS `require('./x')`。

只跟踪相对路径和明确位于工作区内的路径。解析时按现有运行约定尝试显式文件、常用源码扩展和目录入口。裸包名视为 manifest 依赖，不进入 ZIP。

### 6.2 Python

分析相对导入以及可映射到脚本工作区内模块的绝对导入，包括对应 `.py` 文件和包目录的 `__init__.py`。标准库、site-packages 与虚拟环境内容不进入 ZIP。

### 6.3 静态资源

JS/TS 中静态 import 的图片、样式或文本资源可随依赖图加入。代码通过运行时拼接、`fs`/`open()` 等方式读取的资源必须写入 `export.include`。

无法解析的本地静态依赖视为导出错误，防止生成表面成功但无法运行的包。

## 7. 强制排除策略

排除策略优先级高于依赖发现和 `export.include`。至少排除：

- 依赖与环境：`node_modules/`、`.venv/`、`venv/`、`site-packages/`；
- 版本控制与缓存：`.git/`、`.svn/`、`__pycache__/`、`.pytest_cache/`、`.mypy_cache/`；
- 构建与运行产物：日志文件，以及 `dist/`、`dist-electron/`、`release/`、`out/`、`coverage/` 中未被入口依赖图引用的文件；manifest 入口及其静态依赖可位于这些目录，但目录不会被批量扫描或整体导出；
- Autoforge 本地状态：`.autoforge-deps.json`、附件暂存目录、运行输出目录；
- 数据库与常见业务数据：SQLite/DB 文件、CSV、Excel、Parquet、压缩归档及其他明确的数据文件扩展；
- 密钥与本地配置：`.env`、`.env.*`、常见私钥和证书文件。

`autoforge.json` 中只应包含 schema 和依赖声明，不含用户填写的环境变量、运行参数或本地偏好；这些内容存于数据库且不会进入导出工作区。

实现使用规范化相对路径进行检查，并拒绝符号链接，以避免路径穿越或链接到工作区外部文件。

## 8. 导出流程与界面

1. 用户在脚本卡片操作菜单选择“导出 ZIP”。
2. 渲染进程调用新增 IPC，主进程读取并校验 manifest。
3. 导出器构建依赖图、应用显式 include、执行强制排除与路径安全检查。
4. 若必要入口缺失、静态依赖无法解析或 include 非法，则中止并显示可操作错误。
5. 显示保存对话框，默认文件名由脚本名和版本生成并清理非法字符。
6. 使用 ZIP 根目录直接放置 `autoforge.json` 和项目文件，不额外嵌套随机目录。
7. 成功后提示导出的文件数量和保存路径；用户取消保存时不创建文件，也不显示错误。

ZIP 条目使用 POSIX 相对路径，并在写入前再次校验无绝对路径或 `..`。

## 9. Hub 安装与重复更新流程

### 9.1 首次安装

1. 校验 `zipUrl` 和非空 `hubScriptId`。
2. 下载 ZIP 到临时目录，安全解压并定位包根。
3. 校验 manifest 和包结构。
4. 查询 `hubScriptId`；不存在则按现有流程导入。
5. 在同一业务操作中写入脚本记录及 `hubScriptId`。
6. 返回 `installed`，聚焦主窗口并选中新脚本。

Hub 安装路径要求提供 `hubScriptId`；普通本地导入不受此限制。

### 9.2 重复安装

1. 完成下载和包校验后查询 `hubScriptId`。
2. 若存在，显示包含脚本名与当前/待安装版本的确认框。
3. 用户取消：删除临时文件，返回 `duplicate_cancelled`，不广播“安装成功”。
4. 用户更新：将新包复制到同级临时工作区，完成校验后通过目录重命名交换工作区。
5. 使用新 manifest 更新名称、描述、版本、入口、语言、依赖、分类和图标等派生字段。
6. 保留本地 `scriptId`、`hubScriptId`、收藏、归档、运行历史、定时任务、默认环境、环境配置和参数。
7. 成功后返回 `updated`，聚焦并选中原脚本。

若文件替换或数据库更新失败，应回滚到原工作区和原元数据；清理临时目录不能影响原脚本。

### 9.3 并发与状态码

数据库唯一索引处理两个并发请求同时安装同一 Hub ID 的竞态。若首次查询未发现、写入时发生唯一冲突，则重新读取现有脚本并进入重复处理，或返回可重试的重复结果，不能留下孤立工作区。

桥响应使用明确状态：

- `installed`：首次安装成功；
- `updated`：用户确认并更新成功；
- `duplicate_cancelled`：用户取消；
- 现有下载、包校验和导入错误码继续保留。

## 10. 组件边界

- `script-package-exporter`：依赖发现、白名单、强制排除、ZIP 生成；不访问 UI。
- `script-workspace`：安全文件枚举、导入与原子工作区替换。
- `script-registry`：按 Hub ID 查询、首次注册、保留偏好的更新编排。
- `hub-script-installer`：下载、解压、重复检测及返回状态。
- `hub-bridge-server`：HTTP 请求/响应映射，不实现业务去重。
- IPC/preload/renderer：保存对话框、导出触发、结果提示。
- 数据库仓储：字段持久化、查询和唯一约束。

## 11. 错误处理与安全

- 下载继续使用超时并限制 HTTP(S) URL。
- 解压必须防止 ZIP Slip，并设置合理的压缩包总大小、条目数量和解压后大小上限。
- 导出和导入均拒绝符号链接及路径越界。
- 导出清单为空、缺少 manifest 或入口时失败。
- 对话框取消属于正常结果，不记录为错误。
- 更新失败保留原脚本，错误信息指出失败阶段但不泄露敏感文件内容。
- 日志可记录 Hub ID、状态和文件数量，不记录用户配置值或业务数据内容。

## 12. 测试策略

### 12.1 导出单元测试

- JS/TS 静态导入、重导出、字面量动态导入和 require 的递归发现；
- Python 包与相对导入发现；
- README 和显式 include；
- 强制排除优先于 include；
- 裸依赖不复制、缺失本地依赖报错；
- `..`、绝对路径、符号链接和越界拒绝；
- ZIP 条目路径、根目录结构及预期文件内容；
- 数据、依赖、密钥和运行产物不进入 ZIP。

### 12.2 Hub 安装测试

- 首次安装保存 `hubScriptId`；
- 相同 ID 取消后无文件和 DB 变化；
- 相同 ID 更新后保留本地 ID 与所有用户偏好；
- 更新刷新 manifest 派生字段；
- 更新中途失败回滚；
- 不同 Hub ID 可安装内容相同的包；
- 普通本地导入仍允许重复；
- 唯一索引和并发冲突不产生孤立目录。

### 12.3 集成验证

- 从脚本卡片导出、检查 ZIP、重新通过 Hub 安装链路导入；
- 重复安装时确认框的更新与取消分支；
- Hub 桥正确返回 `installed`、`updated` 和 `duplicate_cancelled`；
- 主窗口只在安装或更新成功后选中对应脚本。

## 13. 兼容与文档

- SQLite 迁移对现有脚本无破坏性。
- 旧 Hub 安装记录因没有稳定 ID，首次升级后重装仍可能生成一次副本；从该次安装开始可去重。
- 更新 `docs/script-spec.md`，说明 ZIP 导出白名单、`export.include` 和禁止内容。
- 更新 Hub 桌面端契约文档，声明 `hubScriptId` 为 Hub 安装必填字段及新增响应状态。

## 14. 验收标准

- 用户可将任一合法脚本导出为 Hub 可识别的 ZIP。
- ZIP 不包含下载依赖、本地配置、运行产物或业务数据。
- 动态资源可通过受安全策略约束的 `export.include` 补充。
- 同一 `hubScriptId` 不会生成第二个本地脚本记录。
- 重复时用户可更新或取消；更新保留全部本地偏好和配置。
- 更新失败不破坏现有脚本。
- 新旧本地导入行为保持兼容。
