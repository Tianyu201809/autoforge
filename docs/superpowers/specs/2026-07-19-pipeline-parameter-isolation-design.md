# 流水线脚本参数隔离设计

## 背景

流水线节点目前会在执行时读取原始脚本保存的默认参数，再叠加流水线节点值和运行时参数。这使得脚本在单独运行时保存的参数意外进入流水线，导致两种运行场景相互污染。

## 目标

- 流水线节点参数与原始脚本参数完全独立维护。
- 已有流水线节点缺少参数值时按空值处理，不迁移原始脚本默认值。
- 保留并继续使用流水线自身的节点固定值和按环境运行参数。
- 普通脚本运行逻辑保持不变。

## 数据边界

流水线参数分为两类：

1. `PipelineNode.paramValues`：节点级固定参数，随流水线保存，适用于所有环境。
2. `PipelineMeta.paramsByEnv[envId]`：流水线级按环境保存的运行参数，键使用 `nodeId.paramKey`。

原始脚本的 `savedParams`、`paramsByEnv` 以及 `scriptStore.resolveParamsForScript()` 不属于流水线参数来源，流水线执行时不得读取。

## 运行时解析

流水线节点执行时按以下顺序解析参数：

1. 从节点的 `paramValues` 读取固定值；不存在的字段保持空值/未设置状态。
2. 使用当前环境的流水线 `paramsByEnv` 覆盖同名固定值。
3. 使用本次运行传入的流水线参数覆盖已保存值。
4. 使用节点连线映射结果覆盖目标字段。

连线映射仍然是流水线内部的数据流，不能回写原始脚本保存参数。

## 编辑与保存

- 新增节点时不复制原脚本保存参数，节点固定值初始化为空对象。
- 节点检查器编辑固定值时只修改 `PipelineNode.paramValues`。
- 运行抽屉编辑的值只写入 `PipelineStore.setValues()` 的 `paramsByEnv`。
- 保存流水线、运行流水线都不得调用脚本参数保存接口。
- 删除或修改原始脚本参数不会改变流水线已有参数。

## 兼容与错误处理

- 已有节点中缺少的参数不自动补齐，脚本执行层按脚本 schema 的校验规则处理必填参数。
- 已存在的流水线 `paramValues` 与 `paramsByEnv` 原样保留。
- 不新增数据库迁移；现有 JSON 字段已经能够表达独立数据。
- 参数校验失败时沿用现有流水线节点失败状态和错误展示。

## 验证

- 单元/类型检查覆盖参数解析优先级，确认原始脚本默认值不会出现于流水线参数。
- 运行构建与 lint，确认普通脚本执行链路不受影响。
- 手动验证：修改原始脚本参数后运行流水线，流水线结果保持不变；修改流水线参数后仅流水线结果变化。
